Project Overview

Chocoholics Anonymous is an organization dedicated to helping people addicted to chocolate in all its glorious forms. The software to be created for Chocoholics Anonymous (ChocAn) provides data-processing facilities for the organization. All providers can bill ChocAn by filling in member and respective service information in their terminals. The members receiving services are sent a member report, while the providers are sent provider reports at the end of the week. Managers, also, get a summary report. In addition, all three are given an opportunity to request their respective reports anytime. ChocAn data operators can add, delete, or update member and provider records. The software keeps a provider directory that stores all provider data. Our Use Case Diagram provides a simple representation of these tasks, which are explained in depth in our Use Case Descriptions. In our report, we have listed each team member’s contribution. Glossary terms are also included to clarify the usage of terms used for the scope of this project. Our team has 6 members and our submitter is Jahanvi Kathiriya. Our project will use Java to implement all functionality.

Use Case: Request a Report

Context: A ChocAn Manager requests for a report to be run individually.

Actors: Manager

Main Success Scenario:

  1. The system prompts the manager to enter their PIN.
  2. The system verifies the manager.
  3. The system displays the manager menu for the manager who can select an option.
  4. The manager requests for a report to be run.
  5. The week’s file of services is read.
  6. The report is generated.
  7. The report is printed.
  8. The system returns to the verification screen.

Extensions:

2.a. An invalid PIN was entered.

    2.a.1. The system notifies the manager.

    2.a.2. The software returns to step 1.

3.a. The week’s file of services is empty.

    3.a.1. The system notifies the manager.

    3.a.2. The software returns to step 3.

3.a The system notifies the manager.

3.c The use case resumes at step 1.

Use Case: Bill ChocAn includes Validate Member

Context: Provider needs to bill ChocAn after a health care service is provided.

Actors: Provider

Main Success Scenario:

  1. The provider receives the member’s card and slides it through the terminal’s card reader.
  2. The terminal dials the ChocAn data center, and verifies the number.
  3. The terminal displays the word “Validated” to inform the provider that they may proceed.
  4. The provider keys in the current date.
  5. The provider keys in the appropriate six digit service code corresponding to the service they provided.
  6. The software displays the name of the service associated with the service code.
  7. The software prompts the provider to verify the displayed service.
  8. The software prompts the provider for any comments they wish to add.
  9. The software writes a record to disk recording the current date, date the service was provided, provider and member numbers, the service code, and any comments the provider may have added.
  10. The software looks up the fee associated with the provided service and displays it to the provider.
  11. The system returns to the verification screen.

Extensions:

4.a The provider has entered an invalid date

    4.a.1 The software resumes at step 4.

5.a The provider has entered an invalid service code.

    5.a.1 The software resumes at step 5.

The software is displaying the incorrect service.

    7.a.1 The provider tells the software it is displaying the incorrect service.

    7.a.2 The software resumes at step 5.

8.a The comments total greater than 100 characters.

    8.a.1 The terminal prompts the provider to re-enter their comments.

    8.a.2 The provider re-enters comments.

    8.a.3 The software resumes at step 9

Use Case: Run Main Accounting Procedure includes Generate EFT Data, Generate Summary Report, Generate Member Report, and Generate Provider Report

Context: The timer is running the main accounting scenario.

Actors: Timer

Main Success Scenario:

  1. The system waits until it’s Friday midnight unless the manager sends in a request for reports.
  2. On Friday Midnight or when a request is read, the timer fetches the week’s file of services provided from the ChocAn database.
  3. The timer, then, makes a file for each provider listed. In each file, it lists the name of the provider, with some of their additional information, and for each service provided by the provider during that week, the timer lists the name of the member it was provided to and the service that was provided along with some additional data about the same.
  4. The timer also makes a file for each member listed. In each file, it lists the name of the member, with some of their additional information, and for each service received by the member during that week, the timer lists the name of the provider that provided the said service and the service that was provided, along with some additional data about the same.
  5. Next, the timer makes a summary report for the manager. This file lists every provider to be paid that week, the number of consultations each had, and their total fee for that week. It also lists the total number of providers who provided services, the total number of consultations, and the overall fee total.
  6. The timer writes a record of electronic funds transfer (EFT) to disk to be evaluated later by banking software.
  7. The timer sends each file to the person it concerns.
  8. The timer goes to sleep until the next week or until a request is sent by the manager.

Use Case: Validate Member

Context: The Provider needs to verify the member before providing health care services/before billing ChocAn.

Actors: Provider

Main Success Scenario:

  1. The provider reads in the member’s card through the terminal.
  2. The software verifies the number exists.
  3. The software verifies that the member’s payments are up to date.
  4. The software writes “Validated” to the terminal.

Extensions:

2.a. The number does not exist.

    2.a.1. The software writes “Invalid number” to terminal.

    2.a.2. The software returns to step 1.

3.a. The member’s payments are not up to date.

    3.a.1. The software sets the member’s status to suspended.

    3.a.2. The software writes “Suspended” to terminal.

    3.a.3. The software returns to step 1.

Glossary Terms

  1. Member: An entity that is enlisted to receive services provided by ChocAn.
  2. Member Number: A unique 9-digit number attributed to all providers.
  3. Member Report: A report sent to a member that details the services provided by ChocAn.
  4. Provider: A health care professional who provides services to ChocAn members.
  5. Provider Number: A unique 9-digit number attributed to all providers.
  6. Provider Directory: A file that contains all appropriate six-digit service codes and fees corresponding to a service.
  7. Service Code: A six-digit number that specifies the service provided.
  8. Provider’s Terminal: A digital terminal for providers to handle transactions for any services provided to a member.
  9. Provider Report: A report sent to a provider that details the services they provided to members of ChocAn.
  10. Manager: A person in charge of managing several different providers under ChocAn.
  11. Manager’s Terminal: A digital terminal for managers to handle managerial tasks.
  12. Summary Report: a report, given to the manager, which lists every provider to be paid a particular week, the number of consultations each had, and their total fee for that week. It also lists the total number of providers who provided services, the total number of consultations, and the overall fee total are printed.
  13. ChocAn Data Center: The main computer that holds all the information for ChocAn organization including member data, provider data, transaction data, etc.
  14. Interactive Mode: a mode available in the software that allows records of members and providers to be updated, including the addition of new entities and deletion of any entity that resigned
  15. EFT: an electronic transaction of funds
  16. Timer: a non-human automated entity that runs the main accounting procedure
